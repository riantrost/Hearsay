## Project state (2026-07-18, second session — first-desktop-test UX pass)
- **Rian's first real desktop test found seven UX faults; all seven are fixed and verified live** (two seats driven in-browser against `npm run api` + `npm run dev`; 50 tests green, +4). The findings and their root causes, for the record:
  1. **Scrubber over-served** → it's now a collapsed "⟲ session N" pill expanding to the slider on tap, and absent entirely before history spans sessions (fork pinned: "The scrubber is a pill, not a bar"). The dead one-session slider was most of the "is it even working?" confusion.
  2. **No room for the CM's voice** → events carry an optional `atmosphere` prose field under the one-line canon headline (model + shared mutation + both routes + both forms, reveals included; fork pinned: "Events carry atmosphere"). `MAX_ATMOSPHERE_CHARS = 1200`.
  3. **Always-open amend box** → grace-window testimony now renders as read text with quiet `amend` / `scrawl a mark` affordances; a module-level `openEditors` set keeps a deliberately opened editor alive across the poll's re-renders, and drafts still survive.
  4. **Flow unexplained** → a `?` button opens a role-shaped help overlay (owner loop vs player loop), and an empty world nudges the owner toward the first pin.
  5. **Staged pins vanishing** → `withGhosts` required standing at the present, so any scrub hid the owner's secret layer; owner scaffolding (ghosts + staged) now renders at any viewed session — staging is prep, outside the timeline. Reveal replay is untouched.
  6. **Canon losing to testimony** → the surface now reads canon headline (17px, anchor) → atmosphere → testimony (subordinated, indented) → marks moved *below* the events. The vision's "marks first" prose lost to live feedback; ordering was never a pinned fork.
  7. **Buttons needing two clicks** → store-driven wholesale re-renders could replace a button between mousedown and mouseup, eating the click (worst under two-window testing, where every window-entry click fires the focus refetch). Store renders now defer while a pointer is down and flush a macrotask after release — verified live (old DOM held during press, swap after). Bonus catch: author `display:flex` was overriding the `hidden` attribute, so placebar/pastbar/help-overlay were *always visible*; a `[hidden] { display: none !important }` reset fixes the class.
- Not deployed this session — `npm run deploy` when Rian wants the table on it. Browser-pane note: screenshots timed out throughout (capture issue, not the app); verification was DOM-driven.
- **Disappearing-pins root cause found (Rian's follow-up report, same day): KV `list` is eventually consistent.** `loadCampaignData` assembles the campaign from `list({prefix})`, and a just-written key can be absent from list results for up to ~a minute — so a create → 5s-poll → refetch cycle served a snapshot without the new pin, the client applied it, and the pin vanished (returning quietly once KV caught up). Never reproducible locally: wrangler's dev KV has no lag. Fixed on both sides of the wire: **client** — `ApiStore` keeps a 90s ledger of its own writes (`recentWrites`, keyed kind:id, newest wins) and patches them back into any refetched snapshot before the deep-equal, so a laggy refresh can't un-place a pin, un-write testimony, or wind `currentSession` back; **server** — `ensureRecord` in functions/lib.ts backstops every record-reference route (events→pin, testimony→event, marks→testimony, pins/:pid→pin) with a direct `get` before refusing "no such X", since `get` reads your own writes where they were written. Test delta: +3 (freshness: laggy-snapshot pin survival + quiet, clock never winds back, ledger expiry) = 53 green. The server fallback is typechecked but only exercisable under real KV lag — watch the deployed app for the symptom.

## V1 requirements (the owner's, 2026-07-17)
**Done means: the Frostgrave table runs a real session night on it, from their own phones, with fresh data.**
1. **Recent activity is visible on the map** — pins with recent events read as alive at a glance (recency, open testimony slots). This is the vision's "pins carry visual state" line; no schema needed — it derives from session stamps.
2. **Campaign Managers create a campaign** (name + map image) **and add players whenever they want.**
3. **Players join active campaigns via the campaign's join code.**
4. **Players can share the code onward; the owner approves.** A joiner is a *pending* member who can write immediately, visible only to themselves and the owner; approval makes their posts visible to the table. (Fork: "Membership follows the proposal pattern.")
5. **Sessions update fast on every device, and the app can never pin itself stale.** Server-authoritative data, network-first fetch, refetch on focus, short poll while a pin is open; **no service worker in V1.** (Fork: "V1 is server-authoritative.")

## V1 data model (five entities — simplified 2026-07-17)
```
Campaign   { id, name, mapImageUrl, mapW, mapH, currentSession, joinCode }
Member     { id, campaignId, name, role: owner|player, status: active|pending }
Pin        { id, campaignId, x, y, name, hiddenUntilSession? }
Event      { id, pinId, session, canonLine, participantIds }
Testimony  { id, eventId, memberId, session, text, markText? }
```
Down from seven: **Mark folds into `markText` on Testimony** (the fork's own words — a highlight, not a content type) and **SiteCanon defers out of V1 build scope** (the accretes-from-history design stays settled; it returns post-V1). Everything below Campaign carries a `session` stamp — that column is the scrubber, and recency illumination derives from it.
