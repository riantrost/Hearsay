- **The app can't pin itself stale (roadmap step 4, done 2026-07-17):** freshness discipline lands the "server-authoritative, no service worker" fork. Both sides of the campaign GET now say `no-store` (client `fetch` cache option + server `Cache-Control` header). `ApiStore.refresh` is **quiet when nothing changed** (deep-equal guard: no notify, no re-render, so it never disturbs a half-typed draft) and **loses a race with a local write** (a `writeStamp` bumped on every mutation; a refresh whose fetch started before a write lands is discarded as stale). `main.ts` refetches on window focus and on `visibilitychange`→visible, and **short-polls every 5s only while a pin surface is open and the tab is visible** (`syncPoll` starts/stops the interval from `render()`; a `teardown` AbortController unwinds listeners+timer on re-boot). The pin surface preserves unsent words across the poll's wholesale re-renders: every writable field carries a `data-draft-key`, and value+focus+caret survive the rebuild (`harvestDrafts`/`restoreFocus`); `lineForm` clears optimistically and restores the text on a rejected submit. A standing build gate (`scripts/verify-build.mjs`, wired into `npm run build`) fails the build if any service-worker-shaped file or `serviceWorker.register(` lands in `dist` — Litany's trust-the-proof habit. Verified live: poll fires exactly 2× over 11s while a pin is open, 0× once closed; a draft with focus+caret survived a cross-device write landing mid-type; console clean.
- **The scrubber becomes a place you can stand (roadmap step 7, done 2026-07-17):** viewing the past is signposted and stable. - **Dropping a pin is a deliberate act again (roadmap step 5, done 2026-07-17):** the spike's map UX ported to the Vite line, honoring the "deliberate pin placement" fork the old code had quietly broken (any owner tap on open ground minted a pin). Now the owner **arms** placement — a floating `＋ Pin` button flips to `✕ cancel`, a pulsing placebar reads "Tap the map where it happened", the map cursor becomes a crosshair — and the *next* map tap places one pin (still `prompt`→ghost, surface opens on it), then disarms; outside that mode the map only browses (tap selects/clears). Arming is owner-only and present-only: scrubbing back or a non-owner seat disarms it. Second half: **`--pin-k` counter-scale** — the Viewport gained an `onTransform(scale)` callback that sets `--pin-k = clamp(1/scale, 0.5, 2.4)` on the map-host; pins render via a CSS `transform: translate(..) scale(calc(u * var(--pin-k)))` (not the SVG attribute) so the var re-applies on every zoom frame without a re-render, `transform-box: view-box` + `transform-origin: 0 0` giving the attribute's clean pivot at the pin's own point. Verified live (headless-driver seat against `wrangler pages dev`): un-armed tap places nothing, armed tap creates the ghost + disarms + opens the surface; in the working zoom band `worldScale × pinK = 1.0` so the dot holds a constant ~18px, the clamp intentionally letting it drift only at the extremes (51px maxed-in, 6.5px maxed-out). 25 tests green, `tsc` clean.

## V1 requirements (the owner's, 2026-07-17)
**Done means: the Frostgrave table runs a real session night on it, from their own phones, with fresh data.**
1. **Recent activity is visible on the map** — pins with recent events read as alive at a glance (recency, open testimony slots). This is the vision's "pins carry visual state" line; no schema needed — it derives from session stamps.
2. **Campaign Managers create a campaign** (name + map image) **and add players whenever they want.**
3. **Players join active campaigns via the campaign's join code.**
4. **Players can share the code onward; the owner approves.** A joiner is a *pending* member who can write immediately, visible only to themselves and the owner; approval makes their posts visible to the table. (Fork: "Membership follows the proposal pattern.")
5. **Sessions update fast on every device, and the app can never pin itself stale.** Server-authoritative data, network-first fetch, refetch on focus, short poll while a pin is open; **no service worker in V1.** (Fork: "V1 is server-authoritative.")

## V1 data model (five entities — simplified 2026-07-17)
```
Campaign   { id, name, mapImageUrl, mapW, mapH, currentSession, joinCode }
Member     { id, campaignId, name, role: owner|player, status: active|pending }
Pin        { id, campaignId, x, y, name, hiddenUntilSession? }
Event      { id, pinId, session, canonLine, participantIds }
Testimony  { id, eventId, memberId, session, text, markText? }
```
Down from seven: **Mark folds into `markText` on Testimony** (the fork's own words — a highlight, not a content type) and **SiteCanon defers out of V1 build scope** (the accretes-from-history design stays settled; it returns post-V1). Everything below Campaign carries a `session` stamp — that column is the scrubber, and recency illumination derives from it.
